CloudPulse — план выполнения проекта
Visual Website Monitoring Platform

Общее описание проекта
CloudPulse — это cloud-native платформа для мониторинга сайтов и API.
Пользователь добавляет URL, система регулярно проверяет доступность сайта/API, измеряет latency, сохраняет историю проверок, создает incidents при падениях и показывает состояние через frontend dashboard.

Основной стек:
- Backend: Python FastAPI
- Frontend: React + Vite
- Database: PostgreSQL
- Queue/cache: Redis
- Worker: Python worker
- Scheduler: Python scheduler
- Containerization: Docker + Docker Compose
- Orchestration: Kubernetes через Rancher Desktop
- CI/CD: GitHub Actions
- Cloud deployment: Google Cloud Run или GKE

Общее обязательное требование по ведению журнала прогресса

В проекте должен вестись отдельный файл прогресса, например:

PROGRESS.md
или
PROJECT_LOG.txt

В этом файле после каждого завершенного этапа нужно тезисно записывать, что именно было сделано.

Пример формата:

Этап 1 — Project skeleton
- Создана структура monorepo.
- Добавлены папки backend, frontend, k8s, .github/workflows.
- Создан root README.md.
- Добавлен .gitignore.
- Описана базовая архитектура проекта.

Этап 2 — Backend skeleton
- Создано FastAPI-приложение.
- Добавлен endpoint GET /health.
- Создан requirements.txt.
- Добавлены базовые файлы config.py, database.py, models.py, schemas.py.
- Проверено, что backend запускается через uvicorn.

Правило:
После каждого этапа Codex должен обновлять PROGRESS.md / PROJECT_LOG.txt и кратко фиксировать:
1. Что было создано.
2. Какие файлы были изменены.
3. Как проверить результат.
4. Есть ли ограничения или незавершенные части.

Этап 5 — Website checking logic

Цель:
Добавить основную логику проверки сайтов/API.

Что сделать:
- Создать app/services/checker.py.
- Реализовать функцию:
  check_monitor(db, monitor_id) -> CheckResult

Логика:
- Загрузить monitor из базы.
- Выполнить HTTP GET на monitor.url.
- Измерить latency в миллисекундах.
- Сравнить status_code с expected_status_code.
- Создать CheckResult.
- Обновить Monitor:
  - current_status
  - last_checked_at
- Если check успешен:
  - Monitor.current_status = UP
  - CheckResult.status = SUCCESS
- Если check неуспешен:
  - Monitor.current_status = DOWN
  - CheckResult.status = FAILED
- Обрабатывать:
  - timeout;
  - connection errors;
  - invalid URLs;
  - unexpected status code.
- Использовать httpx.

Ожидаемый результат:
В backend есть функция, которая реально проверяет доступность URL и сохраняет результат.

Проверка:
Пока можно проверить через временный вызов функции или через следующий этап с manual endpoint.

Проверка:

- Запустить check.
- Проверить, что создан OPEN incident.
- Исправить URL или создать working monitor.
- Проверить закрытие incident после успешного check.

Этап 12 — Scheduler service

Цель:
Добавить сервис, который автоматически планирует проверки.

Что сделать:
- Создать app/scheduler.py.
- Scheduler должен каждые 10 секунд:
  - находить active monitors;
  - проверять, пора ли их запускать;
  - если last_checked_at = null, добавить задачу;
  - если now - last_checked_at >= interval_seconds, добавить задачу;
  - отправить задачу в Redis queue.
- По возможности избегать дубликатов задач.
- Логировать действия scheduler’а.
- Обновить docker-compose.yml:
  - добавить scheduler service;
  - использовать тот же backend image;
  - command: python -m app.scheduler.

Ожидаемый результат:
Система автоматически проверяет сайты/API без ручного запуска.

Проверка:
- Создать active monitor.
- Подождать interval_seconds.
- Проверить, что появляются новые CheckResult.
- Проверить логи scheduler и worker.
